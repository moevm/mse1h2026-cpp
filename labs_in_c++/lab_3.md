# Лабораторная работа 3: Исключения  

Исключения позволяют обрабатывать ошибки в runtime, передавая контроль вверх по стеку вызовов. В C ошибки обрабатываются кодами возврата. C++ использует try-catch для перехвата. 

 

Задания 

N=3: (1) тип исключения (runtime_error/invalid_argument/out_of_range), (2) условие броска (zero/divide/overflow), (3) глубина стека (2-5). Хеш определяет аналогично. 

Пример: тип=invalid_argument, условие=zero, глубина=3. 

 

1: Напишите функцию, которая бросает [тип] при [условие] и поймайте в main на глубине [глубина]. 

Пример: если глубина=3, то у вас должно быть написано 3 функции без main, где одна вызывает другую (main -> foo1 -> foo2 -> foo3 и здесь вброс исключения) 

 

2: Иерархия исключений в классах 

Создайте иерархию из трёх классов: Device, Sensor, Controller. Каждый класс имеет метод process(), который вызывает метод process() следующего в цепочке (глубина вызовов = [глубина]). 

Класс Controller (самый глубокий) бросает исключение заданного [тип] при наступлении заданного [условие]. 

Класс Sensor должен перехватить исключение, добавить в него контекстную информацию (например, "Sensor failed: ") и пробросить дальше (throw;). 

Класс Device перехватывает окончательное исключение и выводит полную информацию об ошибке. 

 

3: Множественные catch-блоки и фильтрация 

Напишите функцию process_data(int value), которая в зависимости от value бросает разные типы исключений: 

Если value == 0 — бросает [тип] (основной тип из параметра). 

Если value < 0 — бросает std::logic_error. 

Если value > 100 — бросает std::overflow_error. 

В main создайте цепочку вызовов глубиной [глубина], где самая глубокая функция вызывает process_data со значением, введённым пользователем. 

Обработайте исключения следующим образом: 

Поймайте std::logic_error и его наследников на глубине 1 от main. 

Поймайте std::overflow_error на глубине 2. 

Поймайте заданный [тип] в main. 

Выведите разные сообщения для каждого типа исключения. 

 

4: Function try blocks и конструкторы 

Создайте класс HeavyResource, который: 

Содержит два вложенных объекта, конструкторы которых могут бросать исключения: 

DatabaseConnection db — может бросить std::runtime_error при подключении. 

FileHandler file — может бросить std::ios_base::failure при открытии файла. 

Класс также содержит массив динамической памяти int* data размером [глубина] * 1000 элементов. 

Реализуйте конструктор HeavyResource с function try block, который: 

Инициализирует db и file. 

Выделяет память под data. 

Если при инициализации db или file происходит исключение, поймайте его в function try block, залогируйте и пробросьте дальше, преобразовав в заданный [тип] исключения. 

Важно: Память под data не должна утекать, даже если конструктор бросает исключение. 

В теле конструктора (после инициализации) реализуйте логику, которая может бросить исключение при выполнении заданного [условие] (например, если data не удалось заполнить). 

В main создайте цепочку вызовов глубиной [глубина], где на разных уровнях создаются объекты HeavyResource, и продемонстрируйте, что: 

При исключении из конструктора все уже созданные члены класса корректно уничтожаются. 

Исключение правильно преобразуется и пробрасывается наверх. 

Внешние объекты, созданные до возникновения исключения, также уничтожаются. 

Пример структуры для глубины 3: 

void level3() { 
   HeavyResource res; // может бросить исключение 
} 
 
void level2() { 
   SomeOtherResource x; 
   level3(); 
   AnotherResource y; 
} 
 
void level1() { 
   level2(); 
} 

 

5: RAII-менеджер ресурсов с гарантиями безопасности 

Создайте шаблонный класс ScopedResource<T>, который управляет ресурсом типа T (например, FILE* из C, дескриптор файла, или просто int*). 

Конструктор захватывает ресурс (выделяет память/открывает файл). 

Деструктор гарантированно освобождает ресурс (даже при раскрутке стека). 

Добавьте метод modify(), который: 

При выполнении [условие] бросает исключение заданного [тип].

Может бросить исключение и в других случаях (например, при нехватке памяти). 

Реализуйте правильное копирование (запретить или реализовать глубокое копирование) и перемещение (transfer ownership). 

В main создайте цепочку вызовов глубиной [глубина], где на каждом уровне создаются объекты ScopedResource. При возникновении исключения в любой точке должно происходить корректное освобождение всех созданных к этому моменту ресурсов.
