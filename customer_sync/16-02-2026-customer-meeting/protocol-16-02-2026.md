## Протокол встречи

- **Обсуждение необходимости создания модулей для курса по программированию**
    - тестирование на Си: в кафедральном курсе программирования нет такого модуля, об этом почти не рассказывается студентам
    - введение в C++: программирование на C++ не очень подробно разбирается в рамках курса программирования$ необходимо сделать модуль курса, который будет достаточно «сложным»: широкий спектр тем (отличие языков Си и C++, особенности языка), трудоемкость — не должно быть такого, что курс можно выполнить за вечер
- **Обсуждение необходимости создания единой системы проверки студенческих работ на основе подобной системы для курса RISC-V: это решает следующие проблемы**
    - генерация уникальных заданий и унифицированная настройка этих уникальных заданий;
    - использование форматера, санитиайзера и проч утилит — не нужно вызывать для каждого модуля в его коде на moodle, он будет вызываться централизованно;
    - унификация стандарта компиляции;
    - отсутствие дублирования кода;
- **Постановка общих требований:**
    - уникальность заданий: обязательно для лабораторных работ и желательно для заданий модуля, где это возможно; разным людям должны выдаваться разные задания, и каждый студент получает свой опыт работы с новым модулем, не имея возможности успешно запустить код другого студента у себя;
    - задания придумываются нами по найденным нами же источникам; теоретическая часть будет создаваться итеративно, при согласованиях с заказчиком;
    - в работе над проектом подразумевается создание отдельной системы для курса: генерация заданий и проверка решений осуществляется с ее помощью;
    - создание Dockerfile для возможности проверки системы локально;
    - унифицированный формат вывода тестов для студентов: такой, чтобы они представляли подробную информацию о тесте, и такой, чтобы избежать конфликтов с проверяющей системой;
    - важно формулировать все задания так, чтобы избежать «дурацких» вопросов, чтобы студенту было однозначно ясно, что делать, из текста условия;
    - теоретическое наполнение для курсов должно быть, но в виде ссылки на документацию/главу учебника/прочий внешний источник;
- **Обсуждение технических деталей и ответы на вопросы**
    - используются ли сторонние библиотеки для тестирования: нет;
    - работа системы проверки работ для RISC-V (в двух словах): это python-пакет, который устанавливается внутрь coderunner moodle. Из него импортируются функции проверки. Модуль имеет фунцию генерации init, выдающая уникальное задание, и функцию check, в которую подается ответ студента и которая его проверяет.
    - сложность проекта проверки задач по RISC-V: высокая из-за формата работы с самим кодом, написанном на Ассемблер для RISC-V. Для работы системы лабораторных работ по программированию эти модули не нужны; взять и изменить под нужды проекта стоит входные точки (main.py), Dockerfile, архитектуру «импортов» -- то, как организованы файлы для генерации и проверки лабораторных (минимум 2 файла: файл с аргументами командной строки и реализация класса, наследуемого от базового BaseTaskClass);
    - на момент первой итерации доступ к moodle не потребуется;
    - про тесты: студент тестирует нашу функцию (знает сигнатуру и название), проверка студентом выполняется через простое сравнениерезультатов с ожидаемым; студент формирует некоторый отчет о проведенных тестах, где показывает, какие из его нескольки тестов этой функции были успешными, а какие нет; некоторые функции должны тестироваться по модели черного ящика, а другие по модели серого ящика;
    - для лабораторных по тестированию: стоит придумать набор функций, которые будут внутри пайплайна при сочетании выдавать (условно) уникальную комбинацию каждому студенту; например, зависимость работы от хэша, посчитанного от имени студента;
    - для задач по C++: они должны быть не очень простыми (не просто создание и вывод структуры), допускаются пару простых задач, но лучше создавать задачи под темы, где нужно знать тонкости (перегрузка операторов, лямда-выражения, etc);
    - для лабораторных работ по C++: они должны быть связаны с темы темами, которые будут разбираться в модуле. Чтобы ее решить, студент должен разбираться в темах и идеях (указаны в тз проекта в общем гугл-документе mse).
- **Постановка требований по первой итерации**
    1) создан прототип системы для курса на заглушках;
    2) составлены задания и занесены в гугл-документ заказчика; на момент конца первой итерации требуется, чтобы задания содержали все необходимые темы; необходимо придумать далее, на следующих итерациях, может правиться;
        - стоит сделать упор на идейном наполнении задач (мы хотим для каждого задания создавать уникальный набор аргументов); функции могут быть совершенно разными; единственное требование — реализация функции тестирования студентом не должна быть слишком сложной (до 200 строк);
        - функции по тестированию рекомендуется придумывать креативные: например, есть функция, которая управляет роботом, она принимает 2 аргумеента одного вида, 3 другого, и своеобразно реагирует; Если задачи будут неинтересные и не разнообразные, смысла от их большого количества не будет;
        - для каждого из двух модулей курса стоит придумать по 40 задач или более;
        - для каждой из 2 лабораторных придумать 3 или более идей;
        - реализация на первой итерации не нужна;
    3) придумать унифицированный формат вывода для тестов;
    4) создать Dockerfile;
